一、CGO_ENABLED

作用

用于标识（声明） cgo工具是否可用

意义

存在交叉编译的情况下，cgo工具是不可用的。在标准go命令的上下文环境中，交叉编译意味着程序构建环境中的目标计算架构的标识与程序运行环境的目标设计架构的标识不同，或者程序构建环境的目标操作系统的标识与程序运行环境的目标操作系统的标识不同

小结

我们在宿主机编译的可执行文件，而在 Scratch 镜像运行的可执行文件；显然两者的计算机架构、运行环境标识你无法确定它是否一致（毕竟构建的 docker 镜像还可以给他人使用），那么我们就要进行交叉编译，而交叉编译不支持 cgo，因此这里要禁用掉它

关闭 cgo 后，在构建过程中会忽略 cgo 并静态链接所有的依赖库，而开启 cgo 后，方式将转为动态链接

补充

golang 是默认开启 cgo 工具的，可执行 go env 命令查看

二、GOOS

用于标识（声明）程序构建环境中的目标操作系统

如：

  • linux
  • window

三、GOARCH

用于标识(声明)程序构建环境的目标计算架构

若不设置，默认值与程序运行环境的目标计算架构一致

如：

  • amd64
  • 386

OS X 64 位 darwin amd64

Linux 64 位 linux amd64

用于标识（声明）程序运行环境的目标操作系统

五、GOHOSTARCH

用于标识（声明）程序运行环境的目标计算架构

六、go build

-a

强制重新编译，简单来说，就是不利用缓存或已编译好的部分文件，直接所有包都是最新的代码重新编译和关联

-installsuffix

作用：

在软件包安装的目录中增加后缀标识，以保持输出与默认版本分开

补充：

如果使用 -race 标识，则后缀就会默认设置为 -race标识，用于区分race和普通的版本

-o

指定编译后的可执行文件名称

七 Go 静态编译机制

GO 的可移植性

Go语言的强大之处在于其极佳的可移植性。它的源代码可以在任何平台上编译，生成的二进制可执行文件可以在同种类型的操作系统下运行，不同于如Java诸如此类需要特定运行环境的语言，Go的二进利用了静态链接，不需要任何外部依赖就能运行。

为了实现这样的可移植性，Go标准库为常见的操作系统和硬件架构提供了编译支持。这意味着只需改变编译选项，Go代码就可以轻松地在Windows、Linux、macOS、ARM、AMD64等平台上编译和运行。

GO 的静态链接

Go程序的可移植性大部分归功于其静态链接的特性。静态链接意味着所有程序需求的库都会被包含在生成的二进制文件中。

例如，Go去除了C语言里常见的依赖于glibc的动态链接，而选择直接与操作系统的系统调用接口交互。这使得Go程序不仅在运行环境的选择上有更大自由度，同时，静态链接也使得程序在速度、安全性等方面有显著优势。

CGO 影响可移植性

虽然Go语言强大并且简洁，但也并非万能。一些特定的任务可能需要调用由其他语言写的库，尤其是C语言。这时候就需要使用cgo。

但是cgo给Go带来的问题也很明显，cgo会破坏Go原生的静态链接，导致生成的二进制文件依赖于C库，并对特定的硬件和操作系统产生依赖。
