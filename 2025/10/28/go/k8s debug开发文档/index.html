k8s pod 远程debug开发

k8s pod debug开发调试

调试

日常开发工作中，我们不免需要通过观察程序行为来排查问题，打日志或打程序断点。对于云原生应用来说打日志需要改代码后跑流水线重新打包，更新服务才能生效，效率低下，并且是重复性工作不免使人心烦气躁，所以我们在这里提供了两种办法进行远程调试，各有优劣，请开发者酌情选择合适的调试手段进行调试。

名称 优点 缺点
远程调试边车容器 简单易用，新增一个边车容器，使用nodeport打开暴露服务后，使用ide的远程调试功能连上即可使用 1. 功能简单，仅能打断点调试，不能改代码 2. 不可以调试加密镜像 3. 每次打开nodeport后，要配置防火墙放通端口
nocalhost 可以直接在k8s集群中进行开发，本地改代码即可同步到环境中，开发速度快 前置需要准备的步骤多，配置起来比较麻烦

nocalhost

通常我们将应用安装并运行在环境后，往往会碰到一些问题需要进行排查或调试，这时排查办法往往是基于 日志进行分析。开发阶段往往需要频繁的修改代码，如果每次都需要通过日志做排查或调试，那么我们也需要频繁的在流水线进行打包上传，在环境上更新镜像等操作。

这些步骤十分繁琐，如果你厌倦了这样的调试方法，不妨可以看看这里我们给出的建议，使用nocalhostopen in new window进行开发调试。

nocalhost 是一款面向云原生服务的开发调试工具

  • 直接在 Kubernetes 集群中构建、测试和调试应用程序
  • 提供易于使用的 IDE 插件（支持 VS Code 和 JetBrains），即使在 Kubernetes 集群中进行开发和调试，Nocalhost 也能保持和本地开发一样的开发体验
  • 使用即时文件同步进行开发： 即时将您的代码更改同步到远端容器，而无需重建镜像或重新启动容器。

image：定义了开发容器的镜像。当你进入DevMode时，原始的业务容器会被这个镜像替换。这个镜像应该包含你开发所需的全套环境，例如Go语言编译器、调试器、必要的系统库等。使用一个足够完备的镜像对于顺利开发至关重要。

sidecarImage:Nocalhost通过在业务Pod中注入一个sidecar容器来实现文件同步、端口转发等功能。这个配置指定了所使用的sidecar镜像。通常使用默认即可，除非有特殊要求或内部registry定制。

调试配置与热加载

  • command.debug: ["./run.sh"] 与 debug.remoteDebugPort: 5005
    这是你配置中的一键Debug命令。

    • 当你选择Remote Debug时，Nocalhost会在开发容器中执行command.debug定义的命令。

    • remoteDebugPort: 5005必须与你调试命令中程序监听的端口一致。对于Go应用（使用Delve），你的run.sh脚本可能需要这样启动调试器：

    • 在本地IDE中配置好远程调试（连接到容器的5005端口），即可进行断点调试。

  • hotReload: false

    • 当设置为false时，你修改代码后，通常需要手动重启应用进程才能生效。
    • 如果设置为true，Nocalhost会在代码同步后，自动重启command.run或command.debug中定义的命令。对于Go这类编译型语言，自动重启可能仍需要编译时间，但可以省去手动操作。

文件同步、环境变量与端口转发

sync: null

文件同步配置为空，意味着本地代码修改不会自动同步到远端开发容器。你通常需要配置它来实现实时开发

  • 使用mode: gitIgnore可以基于项目的.gitignore文件来决定同步哪些文件。

  • env: [] 与 portForward: []

    • env: 可以在这里为开发容器设置环境变量，例如GOPROXY等。

    • portForward: 配置端口转发非常有用，它可以将开发容器内的端口映射到本地，方便你直接通过本地地址访问容器内的服务

参数说明：

  • CGO_ENABLED=0：关键设置，禁止使用CGO，确保生成纯静态二进制文件，不依赖glibc等外部C库。
  • GOOS=linux 和 GOARCH=amd64：指定目标为Linux 64位系统。
  • -gcflags="all=-N -l"：关键设置，禁止编译器的内联和优化，这是保留调试信息所必需的。
  • -ldflags="-s -w"：可选，用于减小二进制文件体积（会去除符号表和DWARF调试信息，但上面-gcflags已保留必要调试信息）
